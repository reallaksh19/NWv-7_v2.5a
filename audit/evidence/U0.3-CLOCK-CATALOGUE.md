# U0 — Clock call-site catalogue (Up Ahead)

Snapshot under test: `audit/evidence/frozen/up_ahead_2026-06-24.json`
(sha256 `52e90023b5150af6…`, Python-prefetch schema, 92 items).
Method: `grep` over the U0 surface the plan names — `src/intelligence/*`,
`src/services/upAhead*` + `intelligentUpAheadFetcher.js`, `src/utils/planner*`,
`src/viewModels/useUpAhead*`, plus the date utilities they call (`dateKey.js`,
`dateExtractor.js`, `dateDisplay.js`) — test files excluded, cross-checked against
the runtime harnesses `U0-run-projection.*`, `U0-injection.*`, `U0-clockprobe.*`.

Classification: **injected** (a clock param is threaded in from a caller),
**injectable** (a clock param exists but defaults to the wall clock / no caller
passes it), **hardwired** (reads `new Date()` / `Date.now()` directly, no param).

## Class 2D — Date-KEY convention divergence (the U9-3 hazard)
Two conventions coexist; they name different calendar days for the same instant.
| file:line | Convention | TZ behavior |
|---|---|---|
| `services/upAheadService.js:350-351` (`transformPythonItemsToDisplay`) | `new Date(eventStartAt).toISOString().slice(0,10)` → **UTC** calendar day | TZ-independent |
| `utils/dateKey.js:11-15` (`toLocalDateKey`, used by `plannerStorage.js:35`, `dateAware` `formatDateKey`, `canonicalItemBuilder`) | process-**LOCAL** calendar day | TZ-dependent |
| `utils/plannerStorage.js:65` | `new Date(eventDateKey).toISOString()` — parses a LOCAL key as UTC midnight, re-emits ISO | mixed-convention re-emit (a 2nd U9-3 surface inside the planner) |

→ **Proven by EXECUTION** (`U0-clockprobe.*`): a 02:00-IST event
(`2026-06-25T20:30:00Z`) files under **`2026-06-25`** via the static path but
**`2026-06-26`** via the planner path under `TZ=Asia/Kolkata` (`keysAgree:false`);
under `TZ=UTC` both collapse to `2026-06-25`. Contradicts MODE_MATRIX:56
("Date keys … must stay on `YYYY-MM-DD` local convention"). **Finding I013.**

## Class 3 — Iteration / ordering
All Up Ahead sorts are numeric (`publishedMs` desc, `dateValue`/`date` asc) or
`localeCompare` on `YYYY-MM-DD` strings; `Array.sort` is stable (Node ≥11), and
`uniqByKey`/`Set` preserve insertion order. Deterministic given input order (the
parsed JSON array order). Confirmed: N=10 byte-identity in `U0-run-projection.*`.

## Class 4 — Environment
| Item | Note |
|---|---|
| Node version | this audit ran Node **v22.22.2**; `upahead_refresh.yml` runner Node version unconfirmed → cross-version byte-identity **UNVERIFIED** (same residual as A0). |
| Timezone | the static projection's calendar rollover (`getStartOfTodayMs`, `getDayLabel`, weekly-plan origin) follows the **process/browser TZ**. For a browser user that is their device TZ (correct rollover). The static UTC-slice key (`:350`) is TZ-independent and therefore **disagrees with `toLocalDateKey`** under any non-UTC TZ — see Class 2D / I013. |

## Bottom line
Determinism is achievable for a fixed (snapshot, clock): no RNG, stable sorts,
no shared mutable state across runs. **Two divergence vectors remain:**
(1) the static display path has **no injectable clock** (I012) → audit/benchmark
replay needs a global clock shim, and (2) the **two date-key conventions diverge by
up to a day** for 00:00–05:29-IST events (I013) → wrong-day placement risk handed to
U2.3 / U2.8. The live intelligence path's `asOfDate` injection is sound and
execution-verified.
